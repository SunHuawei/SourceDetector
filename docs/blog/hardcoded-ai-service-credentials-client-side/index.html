How to detect hardcoded AI service credentials in client-side code

The reliable way to detect hardcoded AI service credentials in client-side code is to review what the browser can already see: shipped JavaScript, public config, source maps, and live requests. You are looking for real credential material that appears reusable or operational, not just vendor names, model identifiers, or public publishable keys. In practice, the hard part is not finding suspicious strings. It is proving whether those strings are active credentials, dead build artifacts, or intentionally public integration markers. Good triage starts with visible evidence and stops short of breach claims the browser cannot support.

Short answer: treat hardcoded AI service credentials as a verification problem, not a regex problem. Strong signals combine recognizable credential format, live browser use, or runtime references in first-party code. Weak signals include public app IDs, example strings, dead source-map artifacts, and tokens that look secret-like but are intentionally scoped for client use.

What you can detect

Client-side review can detect provider-facing API keys, bearer tokens, signed request helpers, environment-specific config objects, and source-map artifacts that reveal how an AI integration is wired. It can also show whether the browser appears to call a model provider directly, whether the site uses an intermediate backend, and whether a token is static across sessions.

That is useful because visible runtime evidence is stronger than a random string in a minified bundle. Still, even a convincing token pattern does not automatically prove abuse value. Scope, origin restrictions, quotas, and server-side checks may change the real risk.

Signals that deserve attention

  • Recognizable provider key formats. A vendor-specific prefix or credential structure in first-party code is much stronger than a generic long string.
  • Runtime use in live requests. If the browser sends a stable authorization header or token directly to a third-party AI provider, that is a high-value signal.
  • Source maps that reveal credential handling paths. Original file names, helper functions, and environment loading logic can explain whether a suspicious string is actually part of runtime behavior.
  • Mixed-sensitivity config objects. Teams often ship harmless public keys beside riskier operational values. That mix is where false certainty starts.
  • Cross-session reuse. A token that looks identical across reloads or accounts is usually more interesting than one that expires immediately.

How to verify

  1. Start with shipped assets. Save the exact bundle path, code fragment, and surrounding variable names. Context matters more than the match alone.
  2. Check whether the string appears in first-party runtime paths. A value inside old demo code or comments is a weaker signal than one wired into the application's live request flow.
  3. Inspect browser requests. Look for provider domains, authorization headers, signed payload helpers, or client-generated request metadata that points to direct provider usage.
  4. Compare behavior across sessions. If a value changes per page load or per authenticated session, it may be ephemeral by design. If it stays stable, the risk story changes.
  5. Separate what is observable from what is inferred. Say "browser-visible token reused in provider-facing requests" if that is what you saw. Do not jump straight to "account compromise" unless you have direct proof.
A browser finding becomes credible when the code, request path, and runtime behavior all tell the same story.

Common false positives

  • Public publishable keys. Some vendors intentionally expose identifiers meant only to label a project or initialize a low-risk client flow.
  • Model names and endpoint URLs. Seeing gpt-4, claude, or a provider hostname is not the same as seeing a usable credential.
  • Demo code and mock fixtures. Example strings often survive bundling and look more serious than they are.
  • Dead source-map branches. Old modules, comments, or environment files can describe past integrations with no current runtime role.
  • Ephemeral session tokens. Short-lived browser tokens can still matter, but they do not carry the same triage weight as long-lived provider credentials.

What this does not prove

Client-side evidence does not prove provider account takeover, unrestricted billing abuse, or exposure of sensitive model outputs. It also does not prove a missing backend control. A site may expose a token that is domain-restricted, action-scoped, rate-limited, or useless without a server-side exchange you cannot observe from the browser.

When to escalate to manual review

  • The credential-like value is present in first-party production code and tied to a real runtime path
  • The browser sends it directly to a provider or gateway in a stable, reusable way
  • The signal affects a production environment rather than a test fixture or archived asset
  • You can explain the operational risk clearly without guessing beyond the evidence

FAQ

Does a provider hostname in JavaScript mean credentials are exposed?

No. It may only show that the app references a provider SDK, endpoint, or feature flag. You still need credential material or live runtime use.

Are all client-side AI tokens severe?

No. Some are ephemeral, origin-bound, or intentionally public. The severity comes from what the token can do, not from its length alone.

Why do source maps matter here?

They often reveal original file names and helper logic that explain whether a suspicious string is dead code, debug residue, or an active credential path.

What is the strongest browser-visible signal?

A stable credential reused in first-party code and sent directly to a provider-facing endpoint is usually the clearest reason to escalate.
