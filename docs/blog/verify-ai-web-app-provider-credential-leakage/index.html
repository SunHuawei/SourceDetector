How to verify whether an AI web app leaks provider credentials

The practical way to verify possible provider credential leakage in an AI web app is to inspect what the browser can already see: shipped JavaScript, public config, source maps, and live network requests. You are looking for evidence that a real provider credential, signing secret, or unrestricted token is exposed client-side. A model name, public publishable key, or proxy endpoint alone is not enough. Good verification means separating visible clues from actual credential material, then checking whether the evidence points to direct provider use, safe backend mediation, or a false alarm.

What you can detect

  • Provider API keys or token-like strings embedded in shipped JavaScript or exposed config objects
  • Browser requests sent directly to third-party AI providers instead of a backend-controlled proxy
  • Source maps or debug artifacts that reveal provider SDK setup, environment names, or credential handling paths
  • Client-side code that appears to concatenate, transform, or fetch credential material at runtime

You can often detect meaningful exposure signals from the browser. What you cannot do from those signals alone is prove abuse impact, account compromise, or missing provider-side restrictions.

Why this matters

AI web apps often depend on providers such as OpenAI, Anthropic, Google, Azure OpenAI, or model gateways. When a site uses those services correctly, the browser should usually talk to the site's backend or to a tightly scoped public token flow. When it does not, researchers may see provider-facing requests, long-lived credentials, or artifact clues that deserve review.

The key point is restraint: browser-visible evidence can show that something looks wrong, but it does not automatically prove a breach or exploitable loss. Technical credibility comes from showing exactly what the browser reveals and exactly where the uncertainty starts.

Signals worth checking first

  • Direct provider domains in live requests. Calls from the browser to known AI provider endpoints can indicate client-side integration choices worth reviewing.
  • Secret-like prefixes or headers. A recognizable key format, bearer token, or vendor-specific header in readable code or browser tooling is a stronger signal than a vague string match.
  • Public config with mixed sensitivity. Some apps ship both harmless public identifiers and riskier operational settings in the same object. That mix often creates confusion.
  • Source map references. Source maps sometimes reveal the original file names, SDK wrappers, or feature flags that explain whether the browser is handling provider communication directly.
  • Fallback logic. If client code retries against multiple vendors or regions, it may expose integration details that are useful for triage even when no real secret is present.

How to verify without over-claiming

  1. Review shipped assets first. Look at production bundles, public configuration objects, and source map references. Save the exact file path and surrounding code, not just the matched string.
  2. Inspect browser requests. Check whether the page sends requests directly to a provider, what headers are visible, and whether the request appears user-scoped, ephemeral, or static.
  3. Map visible evidence to architecture. Ask whether the app appears to use a backend proxy, a signed short-lived token flow, or true client-side provider access.
  4. Check restrictions before escalating language. A public-facing key may still be domain-restricted, endpoint-limited, or intended only for low-risk publishable use.
  5. Document what is observable. Record the URL, asset name, request destination, header names, and why the signal appears meaningful. Avoid language that implies confirmed compromise unless you have direct proof.

A concrete example: if you see a request from the browser to a provider endpoint with a stable bearer token reused across sessions, that is stronger than simply seeing an environment variable name in a bundle. The first is evidence of live client-side credential use. The second may only be a code artifact.

Common false positives

  • Publishable keys and public app IDs. Many services expose identifiers that are designed to be public and useless on their own.
  • Example strings or test fixtures. Sample keys in docs, demo code, or mock data often survive the build process.
  • Proxy-looking requests. A request path that mentions a provider does not mean the browser is sending a real provider credential. The site may still proxy and sanitize everything server-side.
  • Dead code in source maps. Old branches, commented snippets, or unused modules can look alarming while having no runtime effect.
  • Short-lived session tokens. Some tokens are intentionally ephemeral and bound to a narrow action window.

What this does not prove

Browser-visible evidence does not prove a provider account is broadly accessible, that billing abuse is possible, or that sensitive model data is exposed. It also does not prove the application lacks compensating controls. Even a real secret-like token may be locked down by origin, scope, quota, project policy, or server-side checks you cannot observe from the browser.

When to escalate to manual review

  • The browser appears to call a provider directly with a stable credential or reusable authorization header
  • You can tie the signal to a specific runtime path rather than a dead artifact
  • The exposed material appears operational rather than merely descriptive
  • The finding affects a production site or production build asset
  • You have enough evidence to describe the risk clearly without guessing about breach impact
