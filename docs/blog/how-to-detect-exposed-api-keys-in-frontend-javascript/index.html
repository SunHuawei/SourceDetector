How to Detect Exposed API Keys in Frontend JavaScript

You can detect potentially exposed API keys in frontend JavaScript by reviewing shipped bundles, checking source maps when they are public, and matching suspicious values to live browser behavior. That part is straightforward. The harder part is deciding whether a visible key is actually risky. Some browser-side keys are meant to be public. Others only look important because of naming. The useful workflow is not “find any string that resembles a key.” It is “find a value, locate its runtime role, and decide what that value enables.” That is where real triage starts.

Short answer: review what the browser actually receives, look for suspicious key-like values inside live JavaScript and public source maps, then verify whether the value identifies a client, authorizes a service, or does almost nothing on its own. A visible string is a signal. It is not automatic proof of a secret leak.

What you can detect

Frontend JavaScript can reveal several useful classes of evidence: project identifiers, publishable keys, mis-scoped tokens, hardcoded environment config, service endpoints, and provider names that would otherwise stay hidden. For a researcher or security reviewer, that browser-visible layer matters because it reflects what real users receive, not what internal documentation claims is deployed.

It also has limits. Frontend evidence tells you what is exposed to the client. It does not tell you, by itself, whether the key is privileged, active, rate-limited, origin-restricted, or operationally dangerous.

Signals worth looking for

  • Key-like strings near provider config: values sitting next to service names, SDK initialization code, or request builders are usually more meaningful than isolated regex hits.
  • Source map references: public source maps can expose clearer variable names, configuration blocks, and dead-code history that minified bundles hide.
  • Runtime request usage: a value that appears in actual request headers, query parameters, or POST bodies deserves more attention than a string that only exists statically in a bundle.
  • Privilege hints: code paths involving billing, model calls, uploads, admin flows, or backend orchestration deserve more scrutiny than basic telemetry or content delivery.
  • Cross-file repetition: the same value appearing in config, request code, and feature-specific modules is usually a stronger signal than a single loose match.

How to verify the finding

1. Start with the shipped JavaScript

Inspect loaded bundles and linked assets before you draw conclusions. You want to know where the value appears, which module references it, and whether the surrounding code makes it look operational or decorative. A key in active initialization logic is more interesting than a key-shaped string buried in comments or legacy chunks.

2. Check whether a source map sharpens the picture

If the site exposes source maps, use them to recover names and flow structure. They often turn a vague string match into something you can reason about: a payment SDK key, an analytics write key, a tenant identifier, or a provider credential that should not have shipped at all. But even here, do not overclaim. Better naming improves confidence. It does not replace validation.

3. Tie the value to a real browser action

Watch what happens when the page loads, authenticates, searches, uploads, or calls a third-party service. A string becomes more credible when it appears in live behavior tied to a meaningful feature. If the value never leaves dead code or unused config, the finding should stay lower confidence.

4. Ask what the key actually does

This is the step many reviews skip. Does the value simply identify a frontend app to a provider? Does it unlock a billable API? Does it appear scoped to a public browser use case? Does the surrounding code imply server mediation even though the key is visible in the client? The answer changes the severity more than the string itself.

Common false positives

  • Publishable keys mistaken for private credentials: some platforms intentionally expose browser-safe identifiers.
  • Environment labels: names such as API_KEY, PUBLIC_TOKEN, or CLIENT_SECRET can be misleading without functional context.
  • Dead integrations: a bundle may still contain old vendor names or leftover strings that are no longer exercised in production.
  • Token-like cache and session values: length and entropy alone do not prove privilege.
  • Copy-pasted examples: sample values, test fixtures, or docs snippets sometimes ship with frontend code by mistake, but they may not correspond to a live secret.

What this does not prove

Finding a key in frontend JavaScript does not prove a breach, backend compromise, unrestricted provider access, or exploitable impact. Public artifact review shows that a value is visible to the browser. That is already useful. But security conclusions should stay bounded until you can explain the value’s role, scope, and likely abuse path.

When to escalate

  • The value is used in live requests tied to expensive, sensitive, or state-changing actions.
  • The key appears to authorize a provider call rather than merely identify a client application.
  • The same value surfaces across multiple code paths, reducing the chance that it is dead residue.
  • The exposure reveals a hidden backend dependency or trust boundary that changes how the application should be assessed.

Methodology

This guide is based on browser-visible review only: shipped JavaScript, public source maps when available, runtime request evidence, and context-based verification. It treats frontend artifacts as signals to investigate, not automatic proof of a vulnerability. Last reviewed: 2026-03-22.

FAQs

Can frontend JavaScript contain real API keys?

Yes. Sometimes they are intentional browser-side identifiers, and sometimes they are exposure mistakes. The difference depends on scope, privilege, and runtime use.

Are public source maps necessary for this kind of review?

No. They help, but you can still learn a lot from loaded bundles, request patterns, and browser-side configuration without them.

Is every visible API key a vulnerability?

No. Some keys are designed to be public. What matters is what they enable, not whether the string exists at all.

What makes a finding stronger?

Runtime use, meaningful feature context, privilege hints, and reproducible evidence across multiple frontend artifacts all raise confidence.
